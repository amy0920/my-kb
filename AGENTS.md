# Agent 共用規則

這份文件是這個知識庫裡所有 Agent 的共同作業規範。
目標是讓不同 Agent 可以接力處理同一件事，彼此交接時不漏上下文、不重工、不中斷。

## 使用方式

- 每個 Agent 開始前先讀這份共用規則。
- 再讀對應角色的規則檔。
- 任務結束時，依照交接模板留下狀態。
- 若任務中途換人，下一個 Agent 先看前一輪的交接內容，再決定下一步。

## 版本控管與交接流程

- 每次修改前，先列出本次會變更的檔案。
- 每次修改後，整理改動內容、影響範圍與下一步建議。
- 若有新增或更新工作流程，優先寫回這份共用規則與對應角色說明。
- 若需要留下進度，請同步更新 `docs/PROGRESS.md`。
- 若需要提交版本，commit message 要清楚描述本次改動，不可只寫 `update`。

## 1. 共通目標

- 先理解現況，再動手修改。
- 只做和當前任務直接相關的變更。
- 盡量保留既有內容與使用者已經做過的修改。
- 讓下一個 Agent 可以接手，而不是只讓當下這一步完成。

## 2. 作業原則

### 2.1 先確認再改動

- 先讀取相關檔案與上下文。
- 不要根據猜測直接大改。
- 若資訊不足，先做最小可行假設，並清楚標示。

### 2.2 變更要小而清楚

- 優先採用最小修改範圍。
- 一次只處理一個明確目標。
- 需要重構時，先確保不破壞現有行為。

### 2.3 不覆蓋他人工作

- 看到不屬於自己這次任務的修改，先保留。
- 若有衝突，先說明衝突點，再決定處理方式。
- 不要為了簡化而重寫別人的成果。

### 2.4 說明要可交接

- 每次完成後都要留下可接續的狀態說明。
- 說明要包含：
  - 做了什麼
  - 為什麼這樣做
  - 還剩什麼沒做
  - 有沒有風險或待確認事項

## 3. 交接規則

每個 Agent 結束工作時，請留下以下資訊，方便下一個 Agent 接手。

### 3.1 必留內容

- 目前任務目標
- 已完成項目
- 修改過的檔案
- 尚未完成的部分
- 已知風險或假設
- 驗證結果

### 3.2 建議交接格式

```md
## Handoff
- Current goal:
- Done:
- Changed files:
- Remaining:
- Risks / assumptions:
- Verification:
```

### 3.3 交接準則

- 如果任務已完成，明確標示完成狀態。
- 如果還沒完成，指出下一步最小可執行動作。
- 如果卡住，說清楚卡點與需要什麼資訊。

## 4. 文件與命名

- 文件名稱要能看出用途。
- 若是全域規則，放在 root 層級。
- 若是某個 Agent 專用規則，名稱中要標示角色。
- 若是交接紀錄，使用固定格式，避免每次長相不同。

## 5. 品質底線

- 不做沒有驗證的結論。
- 不在不確定時假裝已經完成。
- 不留下只有自己看得懂的縮寫或省略。
- 不讓下一個 Agent 重新猜一次上下文。

## 6. 建議的 Agent 分工方式

如果這個知識庫之後要拆成多個 Agent，建議用這種方式：

- `Planner Agent`: 負責拆解任務、定義步驟、決定交接點。
- `Research Agent`: 負責查資料、整理背景、補齊缺口。
- `Implementation Agent`: 負責實作、修改檔案、落地變更。
- `Review Agent`: 負責檢查風險、驗證結果、找出遺漏。

這些角色可以依需求調整，但共用規則應維持一致。

## 7. 最終回報模板

```md
已完成：<簡述結果>
變更檔案：<檔案清單>
驗證：<如何確認>
注意事項：<風險 / 假設>
下一步：<若需要接手，下一個 Agent 要做什麼>
```
